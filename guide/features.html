功能详解 ​

CZero 的每项功能都可在 config.json 中单独开关，并由守护进程按各自的计划触发。本页说明每个功能具体做了什么、如何触发、有哪些保护机制。

缓存清理 ​

为高频应用（微信、QQ、抖音）提供各自独立的清理器，按 app_clean.detect_schedule 的频率触发。

触发前的检测 ​

每次触发时，check 会先判断应用当前状态，避免无谓或有害的清理：

  • 只清理"该清的"应用 —— 通过读取 /dev/cpuset/top-app 与 dumpsys activity（ResumedActivity）识别前台应用；结合 oom_score_adj 判定前台主进程。
  • 前台应用不清理 —— 当某应用正处于前台使用时，本轮跳过它，避免影响正在进行的操作。
  • 游戏前台保护 —— 内置一份常见游戏包名清单，当前台是游戏时不执行清理，避免打断游戏。

每个应用的开关 ​

json
"wechat": { "enabled": true, "enhanced": false }
  • enabled —— 是否清理该应用。
  • enhanced —— 增强模式。开启后清理更彻底（范围更广），默认关闭以求稳妥。

其他应用清理 ​

app_clean.other 处理三大应用之外的缓存，按其 schedule 定时执行（默认每天 03:00）。

总开关与时序屏障 ​

  • 受总开关 general.auto_clean 控制。
  • 受 general.temporal_barrier_days（时序屏障）保护：只清理 N 天以前的文件，保护近期数据；设为 0 表示不启用。

后台压制 ​

按 suppress.detect_schedule 的频率周期性压制在后台运行的目标应用。

工作方式：遍历 /proc 找到目标应用的后台进程并结束它们（SIGKILL），从而释放内存、减少后台耗电。关键保护：

  • 前台应用绝不动 —— 正在前台的应用不会被压制。
  • 识别不到前台就不压制 —— 若本轮无法确定前台应用，则跳过，避免误伤。
  • 压制次数计入统计（basis.prop 的 suppress）。

F2FS 垃圾回收 ​

监控 F2FS 文件系统的脏段数量，超过阈值时执行垃圾回收（GC）。为安全起见，GC 会等待屏幕熄灭后再运行，并限制最长运行时间。

字段含义
dirty_threshold脏段 大于 此值时触发 GC（须大于 clean_threshold）
clean_threshold脏段 小于 此值时视为清理完成并停止
wait_screen_off_timeout等待息屏的超时（秒），到点仍亮屏则本轮放弃
max_runtime_sec单次 GC 的最长运行时长（秒），到时强制停止

为什么要等息屏

F2FS GC 属于磁盘密集操作，放在息屏、设备空闲时执行可以避免影响前台使用的流畅度。

GC 功能需要 /data 为 F2FS 分区。非 F2FS 设备可将 gc.enabled 设为 false，其余功能不受影响。

空文件夹清理 ​

按 empty_folder.schedule 定时清扫残留的空目录（默认每天 04:00）。清扫范围由 list/Emptyfolder/directories.prop 定义，并受白名单 emptyfolder_white.prop 约束。

自定义路径清理 ​

除内置规则外，你可以声明自己的清理路径，由每日任务统一清扫：

文件作用
list/clean_paths.prop自定义要清理的路径列表
list/clean_whitelist.prop清理白名单（保护不被清理的路径）

这些列表可通过 CZeroX 的规则编辑器管理，重装模块时也会被继承。

清理并非直接删除，而是先移入回收站（见下节），默认保留 7 天，期间可完整恢复。

回收站与恢复 ​

为避免误删无法挽回，其他 / 自定义路径清理不做真正的删除，而是把命中的文件移动到回收站保存一段时间，确认无碍后才随过期自动清除。

它是怎么工作的 ​

  • 移动而非删除 —— 每个被清理的文件按原目录结构，镜像到回收站中的对应路径，因此可以原样恢复回原位。
  • 按会话归档 —— 每次清理运行创建一个独立会话目录，命名为 <时间戳>-<PID>（如 20060102-150405-12345），并写入清单 manifest.json，记录每个文件的原始路径、大小、创建时间与过期时间。
  • 保留 7 天 —— 保留期固定为 7 天（不可通过 config.json 调整）。过期的会话在下次清理运行时（或手动 purge）自动清除。
  • 不留空目录 —— 本次运行若未实际移动任何文件，会话目录不会落盘；恢复完成后对应会话目录也会被移除。
  • 不被相册扫描 —— 回收站根目录写入 .nomedia，避免被移入的图片 / 视频出现在媒体库中。
  • 保留修改时间 —— 移动 / 恢复均保留文件原始 mtime，恢复后不会被时序屏障误判为"新文件"。
  • 回收站自我保护 —— 清理规则即使匹配到 /storage/emulated/0，也永远不会清理回收站自身，避免"套娃"。

回收站位于内置存储：

text
/storage/emulated/0/Recycle/<会话>/files/<镜像原路径>
/storage/emulated/0/Recycle/<会话>/manifest.json

管理命令 ​

回收站由 list/customize 二进制提供三个子命令，供 CZeroX 或手动调用（输出均为 JSON，便于程序解析）：

命令作用
customize list列出所有回收会话及其清单（文件数、大小、过期时间），用于展示
customize restore <会话ID>将指定会话内的所有文件原样移回原始位置
customize restore all恢复全部会话
customize purge立即清除所有已过期的回收会话

清单缺失也能恢复

若清理进程被中断导致 manifest.json 缺失，list / 恢复会现场扫描会话目录重建清单，恢复行为不受影响。

统计与日志 ​

  • 统计 —— 各应用的清理次数、压制次数记录在 basis/basis.prop；CZeroX 主页展示今日与累计数据。每天 00:00 由 zero 任务重置当日计数。

  • 日志 —— 开启 general.log 后，所有组件写入单一的每日日志 log/<YYYY-MM-DD>.log，每条带来源标签：

    标签来源
    [定时]调度守护进程
    [检测]前台检测
    [微信] [QQ] [抖音]各应用清理
    [压制]后台压制
    [GC]F2FS 垃圾回收
    [空文件夹]空文件夹清理
    [自定义]自定义路径清理

    仅保留当天日志（zero 任务会删除非今日的日志文件）。

两者默认关闭，可随时开启或关闭。

超级岛通知 ​

清理器在开始与结束时可向 CZeroX 广播事件（CleanEventReceiver），用于在应用侧显示执行进度 / 结果通知（含超级岛）。此行为可由环境变量 CZERO_NO_NOTIFY=1 关闭。
