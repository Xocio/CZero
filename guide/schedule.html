调度模型 ​

config.json 里所有调度都用同一个对象表达：

json
"schedule": { "every": "<ISO8601 时长>", "at": "<HH:MM>" }
  • every —— ISO8601 时长，表示"每隔多久"。
  • at —— 可选，HH:MM（24 小时制，补零），表示一天中的执行时刻。仅当周期 ≥ 1 天时有意义。

守护进程 timer_daemon 直接读取 config.json 并在内存中换算成调度表，没有任何中间派生文件，也不需要你接触 cron 表达式。

every 的取值规则 ​

含义every 写法是否需要 at
每 N 分钟PT2M（每 2 分钟）否
每 N 小时PT4H（每 4 小时）否
每 N 天P1D / P3D是（指定几点执行）
每 N 周P1W / P2W是

格式固定：

  • PT<正整数>M / PT<正整数>H —— 时间分量（分 / 时）。
  • P<正整数>D / P<正整数>W —— 日期分量（天 / 周）。

数字必须是 ≥ 1 的正整数。

两类调度 ​

类型字段特点
频率类app_clean.detect_schedule、suppress.detect_schedule、gc.schedule只用分钟 / 小时间隔，不写 at
定时类app_clean.other.schedule、empty_folder.schedule既有间隔（每几天 / 几周），又有时刻 at

对照示例 ​

every / at含义
PT2M每 2 分钟
PT4H每 4 小时
P1D + 03:00每天 03:00
P3D + 03:00每 3 天的 03:00
P1W + 04:00每周一次，04:00

关键约束 ​

务必遵守

  1. 频率类不要写 at（写了会被忽略）；定时类（天 / 周）必须写 at，否则默认按 00:00 执行。
  2. every 必须是 ≥ 1 的正整数。解析失败时该任务会被跳过，等于该功能不执行。
  3. 写完即生效，无需重启 —— 请用原子写入（先写临时文件再 mv 覆盖）。

多天周期的底层机制 ​

cron 表达式无法精确表达"每 N 天"，因此模块对周期 ≥ 1 天的任务采用时间戳门控：守护进程每天在 at 时刻触发，执行前检查 cron/state 里记录的上次运行时间，距上次不满 N 天则跳过。cron/state 由守护进程自动维护，你只需写好 config.json 里的 schedule 对象即可。
